## 원칙 (위반 시 PR 반려)

1. **신뢰성 최우선.** 법령 원문으로 확인 못 한 수치는 `verifyStatus: '확인필요'` 또는 `'강의기반'` — '확정' 남발 금지. 확정 승격 시 `primarySourceVerified: true` + `history`에 검토 기록 추가.
2. **TDD.** 계산 로직은 실패하는 테스트 먼저 → 구현 → 통과 확인. 기존 vitest 컨벤션(순수 계산 → 룰JSON↔lib 교차검증 → UI 렌더 순) 따름.
3. **디자인 동급.** 기존 Airtable Editorial 토큰·MDX 컴포넌트(`<Tbl>`, `<F id>`, `RuleBasis`) 패턴 유지. 신규 CSS 발명 금지.
4. **법령 API 키는 비밀.** law.go.kr OPEN API의 OC 키는 커밋 파일·facts.json·문서 어디에도 넣지 말 것. 실행 시 환경변수 `LAW_OC`로 받거나 사용자에게 요청. 사용자 노출 링크는 키 없는 공개형 `https://www.law.go.kr/법령/{법령명}/{제N조}` (기존 `lib/law-link.ts` 패턴).
5. 공유 파일(`facts.json`, `chapter-meta.ts`, 룰 JSON) 수정은 충돌 방지 위해 태스크당 한 번에 — 병렬 작업 시 `_staging/` 조각 + `_staging/merge_staging.py` 사용.

## 법령 원문 검증 방법 (T1·T2 공용)

```
검색: https://www.law.go.kr/DRF/lawSearch.do?OC={LAW_OC}&target=law&query={법령명}&type=JSON
  → 법령ID/MST/시행일자 획득
본문: https://www.law.go.kr/DRF/lawService.do?OC={LAW_OC}&target=law&MST={MST}&type=HTML
  → 조문 원문 + 부칙(시행일) 확인
```
절차: fact의 `claim` ↔ 조문 원문 대조 → 일치 시 `verifyStatus: '확정'`, `primarySourceVerified: true`, `confidenceScore` 상향, `history`에 `{date, author, note: '원문 확인(법령명 §조)'}` 추가. 불일치 시 `claim` 교정 + `previousValue` 기재. 시행규칙 별표·고시(간이세액표, 4대보험 요율)는 API로 안 잡히면 국세청/공단 고시 페이지로 확인하고 `sourceUrl` 기록 — 그래도 미확인이면 '확인필요' 유지.
